# PR-14 — Residual correctness and bounded retries

## Outcome

PR-14 adds a shared bounded HTTP retry primitive and applies it to FRED, ALFRED, Yahoo, Stooq, and the live FRED fallback. Only transport failures, HTTP 429, and HTTP 5xx retry. Non-transient 4xx responses and all parse/validation failures fail immediately. The current model description is now consistently eight macro scoring factors plus one independent live-risk overlay.

Champion formulas, weights, freshness rules, 45/55 verdict bands, hysteresis, stress thresholds, and portfolio policy are unchanged. There is no migration.

## Retry policy

- Default: 3 attempts, 10 ms base, 100 ms delay cap, full jitter.
- Hard guardrails: at most 5 attempts and at most 30 seconds for any injected delay cap.
- Injectable `sleep` and `random` make tests deterministic with no real waiting.
- The existing named provider timeout wraps every attempt. After exhaustion, the existing provider failure reason and fallback selection remain unchanged.

## Known limitations

- The policy does not interpret `Retry-After`; it uses its bounded local backoff.
- Current callers are GET-only. The primitive does not clone arbitrary request bodies for replay.
- Retries improve brief-transient tolerance but can add bounded request latency before fallback.
- Staging and production behavior remain unverified because no deployment or remote call was authorized.
